## 三、核查 / 核实结果（以真实数据实算）

针对最关键的「助手文本重建」疑点，直接用 `capture/phaseB_sse.json` 解析：

| 核查项 | 实测结果 |
|---|---|
| 事件总数 | **18** 个（与 MEMORY 声称一致） |
| 事件类型计数 | `SSE_HEARTBEAT:1` `SSE_ACK:1` `FULL_MSG_NOTIFY:1` `STREAM_MSG_NOTIFY:1` `STREAM_CHUNK:9` `CHUNK_DELTA:2` `SSE_REPLY_END:3` |
| `CHUNK_DELTA` 拼接 | `"呀！有什么我可以帮你的吗😊"`（13 字） |
| `STREAM_MSG_NOTIFY` 文本 | `"你"`（助手首段） |
| `STREAM_MSG_NOTIFY+CHUNK_DELTA` 流式拼接 | `"你呀！有什么我可以帮你的吗😊"`（15 字，**仍缺"好"**） |
| `SSE_REPLY_END.brief`（权威文本） | `"你好呀！有什么我可以帮你的吗😊"`（16 字） |
| `SSE_REPLY_END` 三次出现的 `end_type` | `1`（带 `brief`）/`2`（suggest）/`3`（finish）；**仅 `end_type:1` 含 `brief`** |
| `STREAM_CHUNK` 机制 | 携带 `patch_op`（`patch_object:1`=text_block，`patch_type:1`=set，`patch_value.content_block[0].content.text_block.text="好"`）；另有 `tts_content` 走 TTS 流 |

**核查结论**：
- `CHUNK_DELTA` 单独拼接 ≠ 完整助手文本（缺前缀"你好"段）；即便并入 `STREAM_MSG_NOTIFY` 仍缺"好"——因为"好"字来自 `STREAM_CHUNK` 的 `patch_op`。
- **权威完整文本唯一可靠来源是 `SSE_REPLY_END(end_type:1).msg_finish_attr.brief`**。`CHUNK_DELTA`/`STREAM_MSG_NOTIFY` 仅适合实时逐字显示。

## 四、审核发现清单（按严重度）

### P1（高，已落实优化/细化）

**F1 — 逻辑缺陷：原型 `collectAssistantText` 仅拼 `CHUNK_DELTA`，会静默丢失助手文本前缀**
- 证据：`stream-codec.ts` 原 `collectAssistantText` 只累加 `CHUNK_DELTA.text`；`run_validation.ts` / `validate_stream_codec.py` 用 `brief.includes(assistant)` **子串弱断言**——`"你好呀！…".includes("呀！…")` 为 true，7/7 仍 PASS，但重建文本（13 字）比真实 brief（16 字）少 3 字。
- 后果：若把该函数输出用于**记忆/上下文存储**，会存错文本（丢"你好"前缀），且验证无法暴露。
- 已优化：`stream-codec.ts` 新增 `extractBrief()`，`collectAssistantText` 优先返回 `brief` 为权威文本；`run_validation.ts` 断言改为 `canonical === brief` 逐字相等；`validate_stream_codec.py` 同步强化并保留 `[WARN]` 提示流式拼接不完整。
- 复跑：`node run_validation.ts` → 7/7 PASS（canonical 现正确等于 brief）；`python validate_stream_codec.py` → PASS。

**F2 — 需核实（已核实）：`STREAM_CHUNK`×9 与 `SSE_REPLY_END`×3 被完全忽略**
- 证据：见 §三。`STREAM_CHUNK` 的 `patch_op` 是豆包增量构建正文的底层机制；`SSE_REPLY_END` 仅首个（`end_type:1`）携带 `brief`。
- 已细化：`stream-codec.ts` 注释明确「`brief` 为权威、CHUNK_DELTA 仅显示、STREAM_CHUNK 不解析」；验证器补充 `STREAM_CHUNK` 观测计数。生产落地时若需「无 brief 回退」，应实现 `applyPatchOp`（JSON 补丁语义），已记入 §五 P1-7。

### P2（中/低，已落实勘误或记录）

**D1 — 文档矛盾（已勘误）：融合方案 §2.6「发送请求的精确 JSON 未落盘」**
- 事实：`phaseB_sse.json` 首条 `reqBody` 已含完整请求体（验证结论 §2.1 已纠正，但融合方案 §2.6 仍为旧陈述）。
- 已勘误：融合方案 §2.6 改为「请求体已落盘（勘误）」。

**D2 — 文档/代码出入（待细化）：`BRIDGE_EVENT` 常量位置**
- 事实：MEMORY §2.2 表将 `BRIDGE_EVENT="__DOUBAO_EXPORT_BRIDGE_V3__"` 列为 `contracts` 常量，但代码中 `contracts.ts` **未定义**该常量，实际定义在 `dom-hook.ts` 为 `'__DOUBAO_PP_BRIDGE_V1__'`（Doubao-pp 自有桥名，抄的是「桥模式」而非字面常量）。
- 建议细化：生产落地时把 `BRIDGE_EVENT` 收口到 `contracts.ts`（与 `DOUBAO_SELECTORS` 等同级），`dom-hook.ts` 从 contracts 引用，消除出处歧义。

**T2 — 已知风险（原型占位，生产必补）：`dom-hook.ts` 仅为骨架**
- 事实：`createEmptyRequestCache()` 仅返回空对象，未真正「捕获页面请求响应」；`getUIFramework()` 仅返回全局对象，未读取 `conversation/message`。MEMORY §3.2 描述的「扫描 conversation/message，请求槽位捕获」超出当前代码实现。
- 状态：已在 MEMORY §9 P1-6 记录为唯一未实机验证模块（主风险 = 豆包 SPA 改版）。生产落地时必须补全为真实数据捕获逻辑。

**T3 — 死代码（已处理）：`makeDoubaoStreamParser()`**
- 事实：`stream-codec.ts` 导出但未被任何调用方使用。
- 已处理：新版 `stream-codec.ts` 已移除。

**T4 — 已知（原型硬编码）：`getActiveProvider()` 固定返回 doubao**
- 事实：`active.ts` 未真正按运行期 URL/构建变量切换 Provider。
- 状态：生产改造在 MEMORY §9 P0-3（改造 `fetch-hook.ts`）覆盖，需接入 `isDoubaoRoute`/`isDeepseekRoute` 动态选择。
