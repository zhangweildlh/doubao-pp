## 〇、融合说明与基线校正（先讲清两份方案如何融合、哪些事实被实测纠正）

### 0.1 两份方案的互补关系

| 维度 | 用户版贡献 | 实测版贡献 | 融合取舍 |
|---|---|---|---|
| 接口抽象 | 设计了完整的 `PlatformAdapter` 接口（认证/请求/解析/会话/DOM 选择器） | 提出了 `ChatProvider` 解耦 + `fetch-hook` 改造点 | **统一命名为 `ChatProvider`**，吸收用户版的接口方法细节 |
| 反爬立场 | 明确要求"不使用可被检测的技术" | 实机证明"页面内 fetch 自动带签名 = 非检测" | **非检测优先**成总路线；API 逆向仓库降为"仅参照不抄" |
| 真实协议 | 来自公开逆向资料（有误） | 来自 DevTools 实机抓包（权威） | **以实测版为准**，用户版中被实测推翻处已逐条校正 |
| GitHub 选型 | 多引用 API 逆向仓库（free-api/2api，隐含可被检测路线） | 多引用扩展仓库（downloader/export，即非检测范式） | **可移植代码只取自扩展仓库**；API 仓库仅作协议/参数参照 |
| 维护机制 | patch 文件 / 配置驱动 / 同步脚本 | fork + upstream + 机械替换 + 契约门禁 + 定时巡检 | **融合为 fork+脚本+契约门禁+定时巡检**，吸收用户版的配置化思想 |

### 0.2 被实测纠正的三处关键事实（必须更正到融合方案）

1. **对话端点**：用户版写 `/samantha/chat/completion`（源自 API 逆向仓库）。**实测裁决**：真实触发的 SSE 流端点为 `https://www.doubao.com/chat/completion`（`phaseB_sse.json` 确证，`samantha/chat/completion` 计数 0）；`/samantha/chat/completion` 仅在登录态 JS/预取中出现 1 次，属别名/遗留路由。**融合方案采用 `/chat/completion`。**
2. **SSE 事件表述**：用户版沿用我第一轮公开资料的 `2001/2002/2003` 数字事件。**实测裁决**：真实是**命名事件**（`SSE_HEARTBEAT`/`SSE_ACK`/`FULL_MSG_NOTIFY`/`STREAM_MSG_NOTIFY`/`CHUNK_DELTA`/`SSE_REPLY_END`）；`2001/2002/2003` 实为 `content_type` 内部字段值，**并非 SSE 事件名**。**融合方案以命名事件为准。**
3. **请求体来源路线**：用户版直接照搬 `doubao-free-api` 的 `messages:[{content_type:2001}]` + `completion_option` 结构（该仓库是独立 Node 服务端，非网页版真实结构）。**融合方案不采用"自行构造完整请求体"路线**，而采用"**原地增强页面自有请求**"——既非检测，又免去逆向完整 body 结构的高代价。

## 一、反爬优先技术总路线（满足"尽量非检测"的硬性要求）

### 1.1 原则声明

- **首选且默认路线 A（非检测）**：在豆包页面上下文中，拦截并原地增强页面**自己发出**的请求。请求由页面 JS 构造、由页面 JS 自动注入 `a_bogus`/`msToken`/正确 Cookie，扩展只做"在发出前改几个字段"。服务器视角 = 一次完全正常的用户对话请求。
- **备用路线 B（可被检测，默认禁用）**：独立 HTTP 客户端自行重算 `a_bogus`、携带 Cookie 直连。属无头/签名指纹场景，**仅在路线 A 无法覆盖某功能时才启用**，且启用即意味着接受"可被检测"代价。

### 1.2 路线 A 为什么"非检测"——机制证据链

1. 豆包前端 JS 在页面内 hook 了 `window.fetch`（字节系请求拦截器），对发往 `doubao.com` 的请求自动附加 `a_bogus`、`msToken`、`web_id`、`tea_uuid` 等签名/设备参数（实测版 1.4 结论）。
2. 浏览器扩展的 **content script 运行于 MAIN world（`world:"MAIN"`）时，与页面共享同一份 `window`、同一套 fetch hook**——因此扩展在页面内发起/改写的请求，**自动继承页面的合法签名**。
3. 本方案扩展**不自己算任何签名、不自己造设备指纹、不跑无头浏览器**。所有出站请求都是页面原生请求被"轻轻改了一下文本"。
4. 佐证：第三方 `doubao-free-api` 用**随机伪造的 `a_bogus`/`msToken` + 真实 sessionid Cookie** 也能调通网页端点，说明该 web 端点对签名校验宽松；而我们的路线 A 连"伪造"都不需要，用的是页面真签名——稳健性更高。

### 1.3 路线 B 的启用条件与代价（明确约束）

| 启用条件（满足其一才考虑） | 代价 | 处置 |
|---|---|---|
| 需要在**页面未打开**时主动发请求（如纯后台定时任务） | 需 Playwright/Node 重算 `a_bogus`（参考 `lza6/doubao-2api`），无头指纹可被检测 | 默认不启用；若必须，限定在用户明确知晓的范围内 |
| 页面自有请求无法覆盖某字段增强 | 需自建客户端 | 先评估能否改为"改页面 JS 后再让其发"绕过 |

**结论**：Doubao-pp 的核心能力（记忆注入、工具调用、流式回显、UI 注入）全部发生在"用户正在豆包页面对话"的上下文中，路线 A 天然覆盖 100%，路线 B **默认永不启用**。

## 二、DevTools 实测确证事实（取代一切推测，作为融合方案唯一事实基线）

> 来源：本地 360Chrome（Chromium 132）+ CDP + Playwright 连接器，对 `https://www.doubao.com/chat/` 做"未登录落地页抓包 + JS 静态逆向 + 登录态发消息抓 SSE"三步验证。证据见 `capture/`。

### 2.1 真实对话端点（实测裁决 §0.2-1）

```
POST https://www.doubao.com/chat/completion
     ?aid=497858
     &device_id=<19位数字>
     &device_platform=web
     &doubao_device_platform=web
     &doubao_pc_version=3.28.x
     &language=zh&region=CN&sys_region=CN
     &samantha_web=1
     &web_id=<19位数字>&tea_uuid=<同web_id>&web_tab_id=<uuid>
     &version_code=20800&pc_version=3.28.x&pkg_type=release_version&real_aid=497858
     &use-olympus-account=1
```
- `aid=497858` 是豆包网页版固定 app id（与 `doubao-free-api` 的 `DEFAULT_ASSISTANT_ID="497858"`、用户版 §3.2 的 `bot_id` 体系一致，但**路径是 `/chat/completion` 而非 `/samantha/chat/completion`**）。
- `/im/*`、`/alice/*`、`/biz/*`、`/samantha/*` 是会话列表/配置/埋点/资源等**辅助接口**，非对话流主端点。

### 2.2 SSE 流格式（实测帧，命名事件）

完整事件序列（发"你好"实测）：

```
event: SSE_HEARTBEAT      data: {}                                  ← 心跳
event: SSE_ACK            data: {ack_client_meta:{conversation_id, section_id}, ...}  ← 回会话元信息
event: FULL_MSG_NOTIFY    data: {message:{content_block:[{block_type:10000, content:{text_block:{text:"你好"}}}]}}  ← 回显用户消息
event: STREAM_MSG_NOTIFY  data: {content:{content_block:[{block_type:10000, content:{text_block:{text:"你"}}}]}}  ← 首个助手内容
event: CHUNK_DELTA        data: {text:"呀！有什么我可以"}            ← ✅ 逐字文本增量
event: CHUNK_DELTA        data: {text:"帮你的吗😊"}
event: SSE_REPLY_END      data: {end_type:1, msg_finish_attr:{brief:"你好呀！有什么我可以帮你的吗😊"}}  ← 结束+摘要
```

**解析规则（权威，实测校正）**：★ **完整助手文本以 `SSE_REPLY_END(end_type:1).msg_finish_attr.brief` 为准**（与服务端定稿逐字一致）。`CHUNK_DELTA`/`STREAM_MSG_NOTIFY` 仅用于实时逐字显示，二者拼接不等于最终完整文本（缺失 `STREAM_CHUNK` 的 `patch_op` 增量）。`STREAM_CHUNK`（实测共 9 个）携带 JSON 补丁增量（`patch_object` 定位、`patch_type` 定操作、`patch_value` 为值）构建正文，`tts_content` 另走 TTS 流。`2001/2002/2003` 是 `content_type`/`type` 内部字段值，**不是 SSE 事件边界**。（详见《协同审核与优化报告》F1/F2）

### 2.3 消息体结构（实测）

```json
{ "content_block": [ { "block_type": 10000, "content": { "text_block": { "text": "..." } } } ] }
```
- `block_type=10000` = 文本块；`block_type=2074`（实测版记录）= 图片块。
- 用户消息回显（`FULL_MSG_NOTIFY`）与助手增量（`CHUNK_DELTA.text`）共用此 `content_block` 结构。

### 2.4 认证 / 反爬（实测）

| 项 | 实测结果 |
|---|---|
| 登录态 | 字节 Passport 系 Cookie（含 `sessionid` 类凭证）+ 页面 `msToken` |
| 请求签名 | `a_bogus`（字节系，等同 TikTok X-Bogus）+ `msToken` + `web_id`/`tea_uuid`，**由页面 JS 自动注入** |
| PoW | **无**（与 Deepseek 的 `pow.ts` 根本不同） |
| 可被检测风险 | **低**：路线 A 下扩展不发任何自有签名请求 |

### 2.5 DOM 钩子落点（实测）

- 输入框：`textarea.semi-input-textarea`（占位符"发消息…"）。
- 前端框架全局对象：`window.DoubaoUIFramework`（豆包 SPA 框架）——**页面注入与 UI 钩子的天然落点**，等价于 Deepseek-pp 钩 Deepseek 框架的做法。
- 确认来源：`cf-jx/doubao-export-chat-free` 的 `content.js` 首行即 `const UIFramework = window.DoubaoUIFramework || {};` —— 社区已验证此钩子稳定可用。

### 2.6 请求体（实测限制与应对）

- **请求体已落盘（勘误）**：首轮曾误判"请求体未落盘"，后续重读 `capture/phaseB_sse.json` 首条 `reqBody` 确证完整请求体已落盘（见 `validate/验证结论.md` §2.1）。设计抉择不变：路线 A 原地增强仍无需自行构造完整请求体。
- **应对（关键设计抉择）**：因采用"**原地增强**"路线 A，扩展**无需自行构造完整请求体**——只需在 `fetch` 钩子里定位出站 `/chat/completion` 请求体中"用户文本"所在节点，把记忆/Skills 注入进去即可。这把"逆向完整 body 结构"的**高代价**直接降级为"定位一个文本节点"的**低代价**，且对 body 结构小幅变动更鲁棒。
- 若仍想要完整 body 做对照：重跑 `capture/desrive_send.py` 并在 `capture/capture_auth.py` 的 `on_request` 补一行 `postData` 落盘即可（脚本已具备，非新开发）。

### 3.2 可移植（抄代码/范本，逐一详述）

#### ★ 移植源 P1：`LauZzL/doubao-downloader` —— 清单/入口配置范本

- **移植文件**：`manifest.json`（原生 MV3）
- **原文件关键内容**：
  ```json
  {
    "content_scripts": [{
      "matches": ["https://www.doubao.com/chat/*"],
      "js": ["doubao-downloader.user.js"],
      "run_at": "document_end",
      "world": "MAIN"
    }],
    "permissions": ["activeTab"]
  }
  ```
- **移植目的**：作为 Doubao-pp 的**入口与清单配置范本**。
- **移植后实现的功能**：让 content script 在豆包对话页加载，且运行于 **MAIN world**（继承页面 fetch hook，从而获得自动签名——这是"非检测"的技术前提）。
- **与 Deepseek-pp（WXT+React+TS）兼容性**：
  - Deepseek-pp 用 **WXT 框架**，清单由 `wxt.config.ts` 生成，**没有手写 manifest.json**；content script 的 `matches`/`world` 写在各 entrypoint 文件里（如 `entrypoints/content.ts`、`entrypoints/main-world.content.ts`）。
  - **因此"移植"= 把上表的配置值翻译进 WXT**：在 doubao 的 content entrypoint 设置 `matches:['*://www.doubao.com/chat/*']` 与 `world:'MAIN'`，在 `wxt.config.ts` 的 `manifest` 字段补 `host_permissions:['https://www.doubao.com/*']`。结构兼容，仅配置格式需适配。
- **后期维护成本**：**极低**（仅路径/名称字符串，可被 `port-from-upstream.mjs` 机械替换覆盖）。

#### ★ 移植源 P2：`cf-jx/doubao-export-chat-free` —— 页面注入 + UI 钩子 + 事件桥范本（最核心）

- **移植文件**：`src/content.js`（原生 JS IIFE）
- **可移植的四段代码（逐一说明）**：

  | 段 | 代码片段（实测摘录） | 移植目的 | 移植后功能 | 维护成本 |
  |---|---|---|---|---|
  | P2-a 框架钩子 | `const UIFramework = window.DoubaoUIFramework \|\| {};` | 落点豆包 SPA 框架 | 读取/监听会话与消息、挂载注入 UI | 低（钩子稳定） |
  | P2-b 消息状态过滤 | `const HIDDEN_MESSAGE_STATUSES = new Set([1,3,5,7,19]);` | 区分可见/隐藏消息 | 正确识别"正在生成/已隐藏"消息，避免误注入或重复处理 | 低（枚举稳定，随版本微调） |
  | P2-c 事件桥 | `const BRIDGE_EVENT = "__DOUBAO_EXPORT_BRIDGE_V3__";` | content ↔ popup/worker 通信 | 把豆包事件转发给侧边栏/浮窗 React 组件 | 低（桥模式稳定） |
  | P2-d 请求槽位捕获 | `createEmptyRequestCache()` 含 `single/recent/title` 槽位，用于**捕获页面自有请求的响应** | 读取豆包内部 API 返回（如历史、会话元信息）而不自己发请求 | 非检测地获取页面数据，供记忆/导出使用 | 中（依赖页面 API 路径，随改版跟进） |

- **移植目的**：作为 Doubao-pp 的"**页面注入层 + UI 钩子层 + 事件桥 + 请求捕获层**"范本——这正是路线 A"非检测"范式在代码层面的直接体现。
- **移植后实现的功能**：在豆包页面内钩住 `DoubaoUIFramework`、识别消息状态、把对话事件桥接到扩展的 React 侧边栏/浮窗、捕获页面自有请求响应供记忆系统消费。
- **与 Deepseek-pp 兼容性**：
  - 原文件是**原生 JS IIFE**；Deepseek-pp 用 **TypeScript + React + WXT**。
  - **移植 = 重写为 TS 模块**（如 `core/provider/doubao/dom-hook.ts` + `entrypoints/doubao-main-world.content.ts`），把上述四段逻辑接入 Deepseek-pp 现有的 `core/messaging/`、`sidepanel/` 与 `floating-chat`。
  - 逻辑兼容（钩子/枚举/桥/捕获思想通用），**代码风格需适配**（IIFE → TS 模块 + React 组件）。
- **后期维护成本**：**中**——DOM 钩子与请求捕获随豆包 SPA 改版需跟进；但状态枚举与桥模式稳定，可用冒烟测试守护。

#### ★ 移植源 P3：`ihmily/doubao-nomark`（扩展部分 `extension/edge/`）—— 清单最小化印证

- **移植文件**：`extension/edge/manifest.json`
- **关键内容**：`matches` 含 `https://www.doubao.com/chat/*`、`https://www.doubao.com/thread/*`；`run_at:"document_start"`；`world:"MAIN"`；`permissions:["activeTab","storage"]`（最小权限）。
- **移植目的**：与 P1 互相印证"清单最小化 + MAIN world"的正确写法；`document_start` 可作为 doubao 注入时机参考（P1 用 `document_end`，二选一，建议 `document_idle`/`document_end` 视钩子需求而定）。
- **移植后功能**：同 P1（入口配置）。
- **维护成本**：极低（同 P1）。
- **兼容性**：同 P1（翻译进 WXT）。

### 3.3 路线 B 参照（明确排除抄写）：`lza6/doubao-2api`

- 它用 Playwright 无头浏览器重算真实 `a_bogus` 再发请求——是教科书级"可被检测"实现。
- **融合方案立场**：**不抄其任何代码**；仅借其两点事实（① 真实端点别名 `/samantha/chat/completion`；② `a_bogus` 由页面/Playwright 签名、可从响应头 `x-ms-token` 刷新 `msToken`）。这些事实已被我们的路线 A 天然满足，无需复刻其机制。

## 四、Doubao-pp 移植方案（技术路线 / 实现方法）

### 4.1 架构：ChatProvider 抽象（融合命名与方法）

吸收用户版的 `PlatformAdapter` 接口细节，统一命名为 **`ChatProvider`**（与实测版一致），目录下并列实现：

```
core/
├── provider/
│   ├── types.ts            ← ChatProvider 统一接口（契约）
│   ├── active.ts           ← 按构建变量/运行时 URL 选择当前 Provider
│   ├── deepseek/           ← 原 core/deepseek/ 收敛为"一个实现"
│   │   ├── contracts.ts  request-codec.ts  stream-codec.ts  pow.ts  active-client.ts  chat-session.ts
│   └── doubao/             ← 新增：豆包实现（唯一手写成本目录）
│       ├── contracts.ts    ← 端点 / 路由 / 查询参数预算（实测 §2.1）
│       ├── request-aug.ts  ← ★ 路线 A：原地增强页面出站请求（非检测）
│       ├── stream-codec.ts ← 命名事件解析 → 增量文本（实测 §2.2）
│       ├── dom-hook.ts     ← window.DoubaoUIFramework 钩子 + 选择器（P2 范本）
│       ├── chat-session.ts ← 豆包会话 URL 解析
│       └── auth.ts         ← 复用页面签名，仅读 Cookie/会话
```

**ChatProvider 接口（融合双方设计）**：
```typescript
interface ChatProvider {
  readonly id: 'deepseek' | 'doubao';
  readonly webOrigin: string;
  readonly contentScriptMatches: string[];
  readonly hostPermissions: string[];
  matchWebRoute(url: string, method: string, baseUrl: string): boolean; // 路由识别（替换原 matchDeepSeekWebRoute）
  augmentCompletionRequest(body: unknown): unknown;   // ★ 路线 A：原地增强，不重建
  parseSSEStream(resp: Response): AsyncGenerator<StreamEvent>; // 命名事件 → 增量
  resolveChatSessionId(url: string): string | null;
  buildSessionUrl(id: string): string;
  readonly selectors: { inputBox: string; assistantMessage: string; actionButton: string };
  readonly uiFrameworkGlobal: string; // 'DoubaoUIFramework'
}
```

### 4.2 实现方法（分层，全部路线 A / 非检测）

1. **拦截增强层（核心改造）**：`core/interceptor/fetch-hook.ts` 从 `import '../deepseek/...'` 改为 `import '../provider/active'`。钩子逻辑：匹配出站 `/chat/completion` → 调 `active.augmentCompletionRequest(body)` → 在 body 内"用户文本"节点注入记忆/Skills/工具上下文 → 放行原请求（页面自动带签名）。**这是非检测的关键：不改 URL、不改签名、只改文本。**
2. **SSE 解析层**（`doubao/stream-codec.ts`）：按实测 §2.2 解析命名事件，暴露 `AsyncGenerator<StreamEvent>`，`CHUNK_DELTA.text` 顺序拼接为助手文本；接回 Deepseek-pp 现有的流式渲染管道。
3. **DOM/UI 层**（基于 P2 范本）：`doubao/dom-hook.ts` 钩 `window.DoubaoUIFramework`、引用 `HIDDEN_MESSAGE_STATUSES` 枚举、用 `BRIDGE_EVENT` 把事件桥到 React 侧边栏/浮窗；选择器**配置化**（放 `contracts.ts` 或独立 `selectors.ts`），便于豆包改版时单点修改。
4. **认证层**（`doubao/auth.ts`）：**不实现 `a_bogus`/PoW**。仅读取页面既有登录态（Cookie/`msToken`），供必要处透传；所有签名交由页面原生逻辑。

### 4.3 构建配置（WXT 适配）

- 入口：`entrypoints/content.ts`、`entrypoints/main-world.content.ts` 的 `matches` 改为 `['*://www.doubao.com/chat/*']`；MAIN world entrypoint 设 `world:'MAIN'`（Deepseek-pp 已有对应 entrypoint，照搬其写法换域名即可）。
- `wxt.config.ts`：`manifest.host_permissions` 增 `https://www.doubao.com/*`；`web_accessible_resources` 匹配豆包域。
- 构建目标：沿用 Deepseek-pp 的 `wxt build --browser chrome`（360 极速浏览器兼容 Chrome MV3）。

### 4.4 机械移植脚本 `scripts/port-from-upstream.mjs`

- 在 `git fetch upstream && git merge upstream/main` 之后运行。
- 自动应用"机械替换"：包名/品牌名、域名（配置层）、i18n 文案、浮窗资产字符串、storage key 前缀 `deepseek_pp_*` → `doubao_pp_*`。
- **显式不触碰 `core/provider/doubao/*`**（手写维护，避免被覆盖）。
- 附带校验：扫描是否残留 `chat.deepseek.com` 等未替换硬编码。

## 五、后期维护策略与成本测算

### 5.1 同步机制（融合双方）

```
Deepseek-pp (upstream)
      │ git fetch + merge upstream/main
      ▼
Doubao-pp (fork, origin) ── 仅手工维护 core/provider/doubao/* 一个目录
      ├─ 引擎层（memory/skill/tool/agent/ui/background）── 自动同步，零维护
      ├─ core/provider/deepseek/* ───────────────────── 自动同步，零维护
      └─ core/provider/doubao/*  ───────────────────── 唯一手写成本
```

- **契约门禁**：当 upstream 改动 `core/provider/types.ts`，标红提醒"豆包适配器需复查"（唯一不可脚本化、但可被预警的成本）。
- **定时巡检（可选自动化）**：每日 `git -C <Doubao-pp> fetch upstream && git diff --stat upstream/main -- core/provider/types.ts`，有变化即通知。
- 吸收用户版的"配置驱动"思想：`PLATFORM_CONFIG` 把域名/前缀/选择器集中，使大部分同步可被脚本覆盖。

### 5.2 维护成本测算（修正用户版工时）

> 修正点：用户版把"实现豆包请求编码 3-4h、SSE 解析 4-6h"列为高代价；**融合方案因采用"原地增强"（§2.6/§4.2）省去了"完整构造请求体"工作，且 SSE 解析已有实测帧样本，落地更快**。但 DOM 钩子仍须手写且随改版跟进。

**首次移植成本**

| 工作项 | 预估工时 | 难度 | 说明 |
|---|---|---|---|
| ChatProvider 解耦（fetch-hook 改造 + types.ts） | 3-4h | 中 | 一次性，一劳永逸 |
| doubao/stream-codec（命名事件解析） | 3-4h | 中 | 有实测帧，照写即可 |
| doubao/request-aug（原地增强，非完整构造） | 2-3h | 中 | 比"完整构造 body"省时 |
| doubao/dom-hook（基于 P2 范本：DoubaoUIFramework + 选择器） | 3-4h | 中 | 需本地验证选择器 |
| WXT 配置 + 机械脚本 | 1-2h | 低 | 翻译 P1/P3 范本 |
| i18n/品牌/存储键批量替换 | 1-2h | 低 | 脚本化 |
| 测试与调试 | 4-6h | 中 | 含本地 360 浏览器验证 |
| **合计** | **17-25h** | — | 低于用户版 20-30h（因免完整 body 构造） |

**每次同步成本（Deepseek-pp 更新后）**

| 场景 | 工时 | 说明 |
|---|---|---|
| 引擎/Deepseek 实现更新 | 0-0.5h | merge 自动获得 |
| `types.ts` 契约变动 | 1-2h | 手工跟进 doubao 适配器（被门禁预警） |
| 豆包自身网页改版（DOM/接口） | 2-4h | 第三方平台固有成本，与上游无关 |
| **平均每次** | **0.5-2h** | 远低于用户版（因范围更聚焦） |

**年度成本**：假设上游月更 2-3 次 → 年 12-36h（不含豆包改版突发跟进）。

### 5.3 风险评估（更新）

| 风险 | 等级 | 说明 / 缓解 |
|---|---|---|
| 端点/参数错误 | **低（已消除）** | 实测 §2.1 已裁决 `/chat/completion` 与完整参数 |
| SSE 私有协议逆向 | 中→**低** | 实测已抓全帧，解析器可一次写好；日后豆包改帧才需跟进 |
| `a_bogus` 风控升级 | **低** | 路线 A 下页面代签；`doubao-free-api` 经验表明 web 端点对签名校验宽松 |

| 登录态/Cookie 失效 | 低 | 路线 A 复用页面登录态，无需自管账号 |

## 七、与用户方案的交叉对比结论（一张表收口）

| 主题 | 用户版 | 实测版 | 融合版最终立场 |
|---|---|---|---|
| 对话端点 | `/samantha/chat/completion` | `/chat/completion`（实测） | **`/chat/completion`**（实测裁决） |
