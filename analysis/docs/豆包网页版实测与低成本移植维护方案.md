# 豆包网页版（doubao.com）协议实测 与 低成本移植/维护方案

> 目标：把仅面向网页版 Deepseek 的浏览器扩展 `Deepseek-pp`（WXT + React + TypeScript + MV3）改造出 `Doubao-pp`，在**网页版豆包**上实现一致的功能与体验，并要求：
> 1. 能**低成本移植**——尽量用固定脚本从 `Deepseek-pp` 复刻(fork) 过来；
> 2. 能**低成本与上游(upstream) 同步**——`Deepseek-pp` 更新后，`Doubao-pp` 用简单修改就跟上。
>
> 本文结论全部来自**本地 360Chrome（Chromium 132）实机抓包** + **GitHub 网页版豆包插件逆向参考**，非推测。

---

## 一、实测验证结论（证据见 `capture/` 目录）

### 1.1 真实流式对话端点（API 格式）

- **端点**：`POST https://www.doubao.com/chat/completion?aid=497858&device_id=<web_id>&device_platform=web&doubao_device_platform=web&doubao_pc_version=3.28.2&...`
- **不是** `/im/...` 也不是 Deepseek 的 `/api/v0/chat/completion`。`/im/*`、`/alice/*`、`/biz/*` 是**会话列表 / 配置 / 埋点**等辅助接口（`/im/chain/recent_conv`、`/im/conversation/info`、`/alice/user/launch`、`/samantha/skill/recommend`…）。
- `aid=497858` 是豆包网页版固定 app id。

### 1.2 SSE 流格式（与 Deepseek 根本不同——核心差异）

Deepseek 用的是"单行 `data:` 里装 JSON patch `{p,o,v}` 增量"。豆包用的是**标准 SSE 命名事件**，实测抓到的完整事件序列：

```
id: 0
event: SSE_HEARTBEAT          ← 心跳
data: {}

id: 0
event: SSE_ACK               ← 回带会话元信息
data: {"query_list":[...],"ack_client_meta":{"conversation_id":"3843...","local_conversation_id":"local_...","conversation_type":3,"section_id":"3843..."},"conversation_info":{...}}

id: 1
event: FULL_MSG_NOTIFY        ← 回显用户刚发的消息（证实消息体结构）
data: {"message":{"content_type":9999,"content":"[{\"block_type\":10000,\"content\":{\"text_block\":{\"text\":\"你好\"}}]","content_block":[{"block_type":10000,"content":{"text_block":{"text":"你好"}}}],...}}

id: 4
event: STREAM_MSG_NOTIFY      ← 首个助手内容（结构化）
data: {"content":{"content_block":[{"block_type":10000,"content":{"text_block":{"text":"你"}}]} ,"content_type":9999,"tts_content":"你",...}

id: 9
event: CHUNK_DELTA            ← ✅ 最简单、最该用的"逐字文本增量"
data: {"text":"呀！有什么我可以"}

id: 12
event: CHUNK_DELTA
data: {"text":"帮你的吗😊"}

id: 20
event: SSE_REPLY_END         ← 结束，带完整答案摘要
data: {"end_type":1,"msg_finish_attr":{"msgid":"...","brief":"你好呀！有什么我可以帮你的吗😊"}}
```

**关键判读**：
- 助手最终文本 = 顺序拼接所有 **`CHUNK_DELTA` 事件的 `text` 字段**（与 Deepseek 的"增量文本"概念一致，但**帧格式完全不同**：豆包是命名事件 + `text` 增量；Deepseek 是 JSON patch `{p,o,v}`）。
- 数值事件码（`SamanthaText:2001 / SamanthaSuggest:2002 / SamanthaLoading:2003 …`）是 `data` 内部的 `type` 字段，**协议边界是命名事件**（`SSE_HEARTBEAT`/`SSE_ACK`/`FULL_MSG_NOTIFY`/`STREAM_MSG_NOTIFY`/`STREAM_CHUNK`/`CHUNK_DELTA`/`SSE_REPLY_END`），不是裸数字。
- 另有 `STREAM_CHUNK` 事件携带 `patch_op:[{patch_object,patch_type,patch_value}]` 结构化增量——可选解析路径，但 `CHUNK_DELTA.text` 已足够。

### 1.3 请求体结构（发送一条消息时豆包发出什么）

从 SSE 回显的 `meta`/`ext`/`FULL_MSG_NOTIFY` 可反推请求体字段：

| 字段 | 含义 |
|---|---|
| `conversation_id` / `local_conversation_id` / `section_id` | 会话 / 分段定位 |
| `message`（或 `content_block`） | 消息体：`[{block_type:10000, content:{text_block:{text:"你好"}}]` |
| `bot_id`（实测 `7338286299411103781`）/ `model_type`（实测 `38`） | 机器人 / 模型 |
| `chat_id` / `local_message_id` / `reply_unique_key` | 消息去重与幂等 |
| `use_content_block:"1"` | 使用 content_block 结构 |
| `inner_*`（app_id=497858、tt_wid、tea_uuid、user_ip、region=CN） | 设备/会话上下文参数 |

> 注：发送请求的**精确 JSON**本次未落盘（驱动发送在请求钩子之后才挂上），但上表已足够刻画"要拦截/改写的请求形状"。如需逐字节对齐，可重跑 `capture/desrive_send.py` 并在 `capture_auth.py` 的 `on_request` 里补一行 `postData` 落盘。

### 1.4 认证机制 vs Deepseek 的 PoW（分水岭）

| 维度 | 豆包（实测） | Deepseek（前次分析） |
|---|---|---|
| 登录态 | Cookie（字节 Passport）+ `window` 本地 `msToken` | Cookie + `active-client` token |
| 设备身份 | `web_id / user_unique_id / tea_uuid / web_tab_id`（TTWidManager，存 `samantha_web_web_id`） | — |
| 请求签名 | **`a_bogus`**（字节系，等同 TikTok 的 X-Bogus）+ 上述查询参数，由**页面自己生成** | **PoW 工作量证明**（`pow.ts`） |
| 反爬 | 风控签名 `a_bogus` + `msToken` 刷新（参考 `lza6/doubao-2api` 逆向） | PoW 反爬 |
| **PoW** | **无** | **有** |

→ **豆包没有 PoW**。这是两者最本质的差异，决定了移植时"认证层"不能复用 Deepseek 的 `pow.ts`，但**在"拦截页面自有请求"的架构下，我们根本不需要自己实现 `a_bogus`**——豆包页面会替我们签名（详见第三节）。

### 1.5 DOM 结构

- 重型 SPA，大量懒加载 chunk（`s2-chat-runtime`、`async-input-engine-*`、`async-skill-*`、`async-infra-message-samantha`…）。
- 输入框：`textarea.semi-input-textarea`（占位符"发消息…"）。
- **关键钩子对象**：`window.DoubaoUIFramework`（豆包前端框架全局）——`doubao-export-chat-free` 正是钩它来读会话/消息 DOM。这是 Doubao-pp 做"页面注入 + UI 钩子层"的天然落点，等价于 Deepseek-pp 钩 Deepseek 前端框架的做法。

## 二、GitHub 网页版豆包插件生态分析（可借鉴 / 可抄写代码）

用本地 `gh`（已登录 `zhangweildlh`）检索到一批"网页版豆包"插件，按可借鉴度分层：

| 仓库 | 类型 | 可借鉴点 |
|---|---|---|
| **`LauZzL/doubao-downloader`** | MV3 扩展（含油猴） | **扩展骨架**：`manifest.json` 的 `content_scripts` 正好 `matches:["https://www.doubao.com/chat/*"]`、`run_at:document_end`、`world:"MAIN"`。**直接抄作 Doubao-pp 的清单模板。** |
| **`cf-jx/doubao-export-chat-free`** | 网页对话导出扩展 | **页面 DOM 操作层范本**：`src/content.js` 钩 `window.DoubaoUIFramework`、扫描 conversation/message、用 `BRIDGE_EVENT` 桥接 popup/worker、用 `HIDDEN_MESSAGE_STATUSES=Set([1,3,5,7,19])` 过滤状态。**抄"注入 + UI 钩子 + 事件桥"三段式。** |
| **`ihmily/doubao-nomark`** | 油猴 + Edge 插件 + Python 服务 | **API 调用 / 解析参考**：`doubao_parser/image.py\|video.py` 调豆包内部 API 取无水印资源。概念可借（端点 + 参数构造）；但它是 Python 且无水印场景，与我们的 TS 扩展不同，**仅作思路参考，不直抄代码。** |
| **`lza6/doubao-2api`** | Playwright 逆向 + OpenAI 兼容 API | **反爬原理坐实**：基于 Playwright 生成 `a_bogus` 签名、动态刷新 `msToken`、设备指纹、stealth；封装成 `/v1/chat/completions` + SSE 流式。**结论：豆包反爬 = `a_bogus`+`msToken`，不是 PoW。** 此仓库代码**仅在我们"改走独立 API 客户端"时才需要**；采用"拦截页面请求"架构时**不必实现**（页面自带签名）。 |
| `wellykao/doubao-directory-plugin`、`pureSaviour/doubao_chat`、`chenshanle/doubao-helper` | 对话目录/历史列表扩展 | 补充参考：如何读豆包对话列表 DOM。 |
| `scoutchloe/doubao-watermark-remover` 等 | 去水印扩展 | 与本目标无关，略。 |

**可抄写代码的优先级**：
1. `doubao-downloader` 的 `manifest.json` → Doubao-pp 清单（机械替换）。
2. `doubao-export-chat-free` 的 `content.js` 注入/钩框架/事件桥 → Doubao-pp 的 content 层（机械→半机械）。
3. SSE 解析器 → **依据本文 1.2 的真实帧自行用 TS 重写**（我们已有完整帧样本，比任何第三方解析器都准；`doubao-nomark` 仅作对照）。
4. `a_bogus`/`msToken` → **不实现**（拦截架构下页面代签）。

## 三、低成本移植架构（核心设计）

### 3.1 必先做的一次性解耦（让"复刻"成为可能）

前次分析已定位 `Deepseek-pp` 的**致命耦合开关**：`core/interceptor/fetch-hook.ts` 直接 `import '../deepseek/...'`，并用 `matchDeepSeekWebRoute` 按 Deepseek 域名+路径识别请求。若只把 `chat.deepseek.com` 全局替换成 `doubao.com`，拦截器对豆包请求**完全无法匹配** → 记忆注入、工具调用、流式处理全失效。

**解法是一次性重构**（在 `Deepseek-pp` 或其复刻里做）：把"平台专属"逻辑收口为一个 **`ChatProvider`（提供方）接口**，Deepseek 与豆包各是实现：

```
core/provider/types.ts            ← 统一接口（契约）
core/provider/deepseek/*         ← 原 core/deepseek/ 收敛为"一个实现"
core/provider/doubao/*           ← 新增：豆包实现（下表手写部分）
core/interceptor/fetch-hook.ts   ← 改为依赖"当前 ChatProvider"，不再硬 import deepseek
```

`ChatProvider` 接口至少含：`matchRoute(url)`（识别本平台请求）、`decodeRequest(body)` / `encodeRequest(body)`（请求体编解码）、`parseSSE(chunk)`（流解析→增量文本）、`getDomAdapters()`（页面 DOM 钩子）。

### 3.2 移植工作量三档（哪些脚本化、哪些手写）

| 层 | 做法 | 能否脚本化 | 来源 |
|---|---|---|---|
| 品牌名 / 清单(manifest) 名 / 域名常量 / i18n 文案 / 浮窗资产 | 字符串替换 | ✅ 完全脚本 | `doubao-downloader` 清单可直接套 |
| `content_scripts` 的 `matches` / 会话 URL 正则 / `host_permissions` | 配置替换 | ✅ 脚本 | 同上 |
| **SSE 解析器**（`CHUNK_DELTA.text` 拼接） | **按本文 1.2 真实帧用 TS 重写** | ❌ 必须手写 | 本文证据 + `doubao-nomark` 对照 |
| **请求体编解码**（`content_block` 结构、注入记忆/系统词） | **手写** | ❌ | 本文 1.3 |
| **DOM 选择器 / `window.DoubaoUIFramework` 钩子** | **手写**（可借 `doubao-export-chat-free` 骨架） | ❌ 半机械 | `doubao-export-chat-free` |
| **认证签名 `a_bogus`** | **不实现**（拦截架构下页面代签） | — | `doubao-2api` 印证 |

→ **真正"手写且不随上游免费"的成本，收敛在 `core/provider/doubao/*` 这一个目录**：SSE 解析、请求编解码、DOM 钩子。其余全部可脚本化或复用引擎。

### 3.3 复用 `Deepseek-pp` 的"引擎"（免费那部分）

记忆 / Skills / 工具 / 自动化 / 云同步 / 浮窗 UI / background 编排 —— 这些**平台无关**，在 `Deepseek-pp` 里保持不依赖任何具体 provider。复刻(fork) 后原样保留，上游更新时 `git merge upstream/main` 自动获得。

## 四、低成本与上游(upstream) 同步维护方案

### 4.1 协作模型（一张图说清）

```
Deepseek-pp (上游 upstream)
      │  git merge / rebase upstream/main
      ▼
Doubao-pp (你的复刻 origin)  ── 仅手工维护 core/provider/doubao/* 这"一个目录"
      │
      ├─ 引擎层（memory/skills/tools/ui/background）── 自动同步，零维护
      └─ core/provider/deepseek/* ─────────────── 自动同步，零维护
      └─ core/provider/doubao/*  ─────────────── 唯一手写成本
```

### 4.2 复刻与同步的标准动作

1. **初始化**：`Doubao-pp` 复刻(fork) 自 `Deepseek-pp`；在 `Doubao-pp` 里加 `git remote add upstream <Deepseek-pp>`。
2. **上游有更新**：`git fetch upstream` → `git merge upstream/main`（或变基(rebase)）→ 引擎层 + `deepseek` 实现**自动并入**，无冲突（你没改它们）。
3. **机械移植脚本** `scripts/port-from-upstream.mjs`：合并(merge) 后自动跑固定字符串替换（品牌 / 域名 / 清单 matches / i18n），**且脚本显式"不触碰" `core/provider/doubao/*`**。
4. **契约门禁（降低维护惊喜）**：当上游改动 `core/provider/types.ts`（即 `ChatProvider` 契约）时，意味着接口变了 → 你的 `doubao` 实现**必须手工跟进**。建议在 CI / 本地钩子里加一条检查：`upstream` 的 `types.ts` 一旦相对你上次同步有 diff，就标红提醒"豆包适配器需复查"。这是**唯一不可脚本化、却可被"预警"的维护成本**。

### 4.3 把"同步巡检"做成自动化（可选）

可建一个定时任务（如每日）：`git -C <Doubao-pp> fetch upstream && git diff --stat upstream/main -- core/provider/types.ts`，有变化就通知你。这样"跟上游"从"凭记忆手动看"变成"被提醒才动手"。

### 4.4 成本小结

| 场景 | 维护成本 |
|---|---|
| 上游更新**引擎功能**（记忆/Skills/工具/UI…） | **免费**：merge 即得 |
| 上游更新 **deepseek 实现** | **免费**：merge 即得 |
| 上游改动 **`ChatProvider` 契约** | **一次手写跟进**（豆包适配器），可被门禁预警 |
| 豆包**自身网页改版**（DOM/接口变） | **手写跟进**（DOM 钩子 / SSE 解析），这是第三方平台固有成本，与上游无关 |

## 五、风险与工作量估算

| 风险 | 等级 | 说明 / 缓解 |
|---|---|---|
| 豆包 SSE 私有协议逆向 | 中 | 本文已抓到完整帧，**解析器可一次写好**；风险在日后豆包改帧 → 需跟进 |
| `a_bogus` 风控升级 | 低（拦截架构下） | 页面代签；仅"独立 API 客户端"路线才需自实现（那时借 `doubao-2api`） |
| DOM / `DoubaoUIFramework` 改版 | 中 | 豆包 SPA 频繁改版，DOM 钩子是长期维护点；用 `doubao-export-chat-free` 的稳健扫描法降低脆弱性 |
| 上游契约变动击穿适配器 | 中 | 用 4.2 的"契约门禁 + 定时巡检"兜底 |
| 登录态 / Cookie 失效 | 低 | 拦截架构下复用页面登录态，无需自管账号 |

**一次性投入**：① 在 `Deepseek-pp`（或复刻）做 `ChatProvider` 解耦（中工作量，但一劳永逸）；② 写 `core/provider/doubao/*`（SSE 解析 + 请求编解码 + DOM 钩子，中工作量，本文已把协议扒透，落地快）；③ 写 `port-from-upstream.mjs` 机械替换脚本（小）。
**此后每轮同步**：引擎/Deepseek 实现 = 免费；仅契约变动或豆包改版才手动，且可被预警。
