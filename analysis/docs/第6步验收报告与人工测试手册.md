## 二、第 6 步核心修复：跨世界中继回退（360Chrome 兼容）

### 2.1 问题背景
第 5 步 `loadInjectionContext()` 在 **MAIN world**（`document_start` 注入的 `main-world.js`）中直读 `chrome.storage` 以装配动态上下文（记忆 local / 技能 sync / MCP local）。但第 3 步已实测确认：**360Chrome 的 MAIN world 不提供 `chrome.runtime`**。若其 MAIN world 同样不提供 `chrome.storage`，则动态注入会退化为空串——功能静默失效（虽 fail-open 不崩溃，但记忆/技能/MCP 注入形同虚设）。标准 Chrome 不受影响。

### 2.2 修复方案（双路径）
重写 `core/provider/doubao/injection.ts` 的 `loadInjectionContext`：

- **快速路径（标准 Chrome）**：`isAuthed()` 且 `hasChromeStorage()`（检测 `chrome.storage.local.get` 存在）→ 直读 `readInjectionContextString()` 合并记忆/技能/MCP。
- **中继路径（360Chrome 等 MAIN world 无 storage）**：`requestContextViaRelay()` 经 `window.postMessage` 向 ISOLATED world 的 `relay.js` 发请求（`CTX_REQ_CHANNEL='__doubaoPpCtxReq'`，带随机 `reqId`）；`relay` 在 ISOLATED world 读 `chrome.storage` 后经 `CTX_RESP_CHANNEL='__doubaoPpCtxResp'` 原路返回；MAIN world 监听匹配 `reqId` 的响应并 resolve 合并后的上下文。

### 2.3 fail-open 不变量（永不 reject）
`requestContextViaRelay` 内置三重守卫，保证 `loadInjectionContext` 在任何异常下都 **resolve（绝不 reject）**，返回空串而非抛错：
1. **存在性守卫**：`typeof window.addEventListener !== 'function' || typeof window.postMessage !== 'function'` → 直接 `resolve('')`。
2. **调用守卫**：`addEventListener` / `postMessage` 调用包 `try/catch`，catch 内 `resolve('')`。
3. **超时守卫**：`setTimeout(RELAY_TIMEOUT_MS=1000)`，超时未收到响应 → `resolve('')`。

`relay.content.ts` 侧同样兜底：读取 `chrome.storage` 出错时仍回传空上下文载荷，防止 MAIN world 悬挂在等待中。

### 2.4 涉及文件
- `core/provider/doubao/injection.ts`：新增 `CTX_REQ_CHANNEL`/`CTX_RESP_CHANNEL`/`RELAY_TIMEOUT_MS`、`hasChromeStorage()`、`readInjectionContextString()`、`buildContextResponse()`，重写 `loadInjectionContext` + 内部 `requestContextViaRelay()`。
- `entrypoints/relay.content.ts`：新增第二个 `window.addEventListener('message', ...)` 处理 `CTX_REQ_CHANNEL` 回读。
- `tests/injection-relay.test.ts`（新增，4 用例）：中继请求 / reqId 严格匹配防串扰 / 纯单测环境 fail-open / relay 侧合并。

### 2.5 回归陷阱（已修）
初版 `requestContextViaRelay` 在最小 `window` 桩（无 `addEventListener`/`postMessage`）的集成测试环境抛错 → Promise reject → 增强被跳过、哨兵未注入、并发 reqId 断言失败（integration.fetch-hook 2 例）。加上存在性守卫 + try/catch 后复检全绿。

## 四、人工真机验收手册（沙箱不可执行，需用户在真实环境完成）

> 适用环境：已登录豆包网页版的 **标准 Chrome** 或 **360Chrome**（扩展均已通过 §三 门禁）。
> 前置：加载 `dist/chrome-mv3`（开发者模式"加载已解压的扩展程序"）；打开 `https://www.doubao.com/chat/`。

### 用例 A — 记忆注入可见（验收项①）
1. 在扩展 popup 的"记忆"标签添加一条测试记忆（如："用户偏好用中文回复"）。
2. 在豆包发起一次对话，打开浏览器 DevTools → Network，筛选 `/chat/completion` 请求。
3. **预期**：请求体 `messages[0].content_block[0].content.text_block.text` 开头含稳定哨兵 `CONTEXT_SENTINEL`，其后紧跟第 5 步装配的动态上下文（含刚才添加的记忆文本）。
4. **判定**：能看到记忆文本被写入请求体 ⇒ 注入生效（标准 Chrome 走快速路径；360Chrome 走中继路径，表现一致）。

### 用例 B — SSE 回显正确（验收项②）
1. 发起对话，观察浮窗与豆包页面回复。
2. **预期**：
   - 浮窗实时流式显示助手文本（`bridgeEmitPage` 派发 `STREAMING_TEXT`）。
   - 最终定稿以 `SSE_REPLY_END(end_type:1).msg_finish_attr.brief` 为准，与豆包页面最终回复一致。
3. **判定**：浮窗实时流与最终定稿均正确 ⇒ SSE 解析链路正确。

### 用例 C — 浮窗多轮正确（验收项⑤）
1. 连续发起 3 轮对话（不刷新页面）。
2. **预期**：每轮浮窗只显示当轮回复，互不串扰；快速连发时 `floating-state` 的 `currentRequestId` 多轮防护生效，旧轮残留不会污染新轮。
3. **判定**：多轮各自独立、无串扰 ⇒ 浮窗多轮正确。

### 用例 D — 技能+MCP+云同步生效（验收项④）
1. 打开浮窗"技能/MCP"标签。
2. **预期**：列出已启用的技能与 MCP 工具（schema 作为上下文注入）；显示云同步状态（Chrome 账户 `chrome.storage.sync` 同步=开启）。
3. **判定**：技能/MCP 列表与同步状态显示正确 ⇒ 第 5 步装配在真机生效。

### 用例 E — 360Chrome 专项（验收项⑥回退）
1. 用 **360Chrome** 加载扩展，重复用例 A。
2. **预期**：即使 MAIN world 无 `chrome.storage`，记忆仍能被注入（走 `relay` 中继回退）；扩展不报错、对话正常。
3. **判定**：360Chrome 下记忆注入仍可见 ⇒ 中继回退在真实环境成立。
